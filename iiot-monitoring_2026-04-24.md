# Анализ и статус проекта IIoT Monitoring (Mobile)
**Дата:** 24 апреля 2026 г.
**Статус синхронизации:** Синхронизировано с System Core v3.5 (TimescaleDB + .NET 10)

## 1. Реактивность и Real-time (SignalR / Postgres Notify)

### Что реализовано:
- **Стрим данных (SignalR)**: Интегрирован `signalr_netcore`. Подписка на `ReceiveMetrics` в `CalculatedDeviceNotifier` обеспечивает отображение данных в реальном времени (latency < 100ms в локальной сети).
- **Валидация типов**: UI корректно обрабатывает типы `ANALOG`, `DIGITAL` и `VIRTUAL` (расчетные датчики на стороне API), согласно схеме `sensor_settings`.
- **Индикация связи**: Виджеты отслеживают статус `system_status` (ONLINE/DEGRADED), транслируемый из БД через API.

### Актуальные задачи (Sync с Системой):
- **Интеграция Deadband (UI-Side)**: Синхронизировать отображение с `deadband_threshold` из `system_config`. Если значение в "мертвой зоне", UI не должен анимировать микро-скачки.
- **Поддержка ADAM-6017 (0-10V / 4-20mA)**: Добавить в `ExpandedSensorCard` отображение сырых данных (raw_value) параллельно с масштабированными (value), так как это критично для пусконаладки.

## 2. Аналитика и Исторические данные (TimescaleDB Integration)

### Состояние:
- **Архитектура**: Система перешла на материализованные агрегаты `metrics_hourly`. Мобильное приложение должно использовать их для "длинных" графиков (>48ч).

### Внедрение (Priorities):
- **Hybrid Charting**: Реализовать логику выбора источника в Flutter-клиенте:
    - Запрос за последние 24 часа -> Raw Metrics (`metrics`).
    - Запрос за неделю/месяц -> Агрегаты (`metrics_hourly`).
- **Performance**: Использовать `fl_chart` с оптимизацией количества точек (Downsampling) для предотвращения лагов на мобильных устройствах.
- **Экспорт**: Добавление генерации PDF-отчетов по сменам (используя данные из TimescaleDB).

## 3. Безопасность и Сеть (Cloudflare Tunnel)

### Статус:
- **Zero Trust**: Приложение настроено на работу через Cloudflare Tunnel. 
- **Фикс нестабильности**: В соответствии с `fixing-cloudflare.md`, клиент форсирует HTTP/2 для стабильной работы через прокси/туннели.

### Что добавить:
- **Session Tracking Awareness**: Добавить обработку ошибки "Max Sessions Reached" (лимит 10 IP на Gateway), чтобы оператор понимал, почему соединение отклонено.

## 4. Система алармов и Квитирование (Critical)

### Что реализовано:
- **Evaluation**: Базовая логика `SensorEvaluator` (Normal/Warning/Critical).

### План на спринт:
- **Push-уведомления (FCM)**: Т.к. API больше не делает `UseHttpsRedirection` (SSL терминация на Cloudflare), настроить корректные Webhook-адреса для уведомлений.
- **Журнал событий (Audit Trail)**: Реализовать экран логов, который тянет данные напрямую из системных таблиц Postgres, включая действия по изменению настроек (Zod-валидированные формы на Web должны отражаться в логах Mobile).
- **Acknowledgment**: Кнопка "Квитировать" (Подтвердить) для операторов. Это действие должно писать в БД статус подтверждения, который затем через SignalR снимает индикацию Alarm на всех клиентах (Web и Mobile).

## Резюме по стеку:
- **State**: Riverpod 3.x (Generator-based) — *Ок*.
- **Database**: PostgreSQL Binary COPY поддержка не требуется (только для Collector), Mobile работает только через REST/SignalR.
- **Ближайшая цель**: Замена заглушек в `LogsScreen` на реальный поток событий из БД и внедрение гибридных графиков.
